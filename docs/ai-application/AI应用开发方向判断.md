# AI 应用开发方向判断
> 这份文档先不急着展开技术细节，  
> 而是把当前最重要的几个判断固定下来：  
> 对传统 Web 开发者来说，AI 方向最现实的切入点是什么？  
> 这个方向到底应该如何定义？  
> 后续又应该怎样按同一套方法论继续拆分知识？
>

---

## 先约定这份文档写给谁
这份文档默认写给：
- 有传统 Web 开发经验
- 做过一些简单的 AI 接口调用或小项目
- 但还没有系统整理过 AI 应用开发的人

所以这里会尽量用“做产品、做系统”的视角来解释，
而不是默认你已经熟悉算法研究、模型训练或复杂 Agent 理论。

这里也顺手固定一个边界：
- 这份文档负责判断“方向是否成立、范围怎么划、主线先放在哪里”
- 它不负责替代后面的核心原则、能力地图和演进线文档
- 它也不负责规定某个具体项目的实施排期

---

## 先说当前结论
**对于有传统 Web 开发背景的程序员来说，当前最现实、最容易形成竞争力的方向，不是模型研究，而是 AI 应用开发。**

这里说的 AI 应用开发，指的是：
- 把大语言模型（`LLM`）能力接入真实产品
- 让 AI 能力和业务流程、数据、权限、前后端交互、监控、安全、成本控制结合起来
- 做出真正能被用户使用、被团队维护、被系统承载的 AI 功能

换句话说，当前更适合把自己定位成：
**会把 AI 能力接入真实产品的工程师**，而不是先把目标定成“做底层模型的人”。

---

## 为什么当前不把目标直接定义成 Agent 开发
这是当前最容易产生误解的地方。

Agent 很热，也很容易成为 AI 方向的代表词。
但如果一开始就把整个方向缩成“Agent 开发”，范围会被压得过窄。

因为现实里的很多 AI 产品能力，其实并不一定需要重 Agent：
- 智能问答
- 智能搜索
- 摘要生成
- 文案生成
- 表单辅助填写
- 文档理解
- 客服 Copilot
- 工作流自动化

这些能力很多时候通过下面这些方式就能做得很好：
- Prompt 设计
- Structured Output
- RAG
- 工具调用（`Tool Use / Tool Calling`，很多资料里也写作 `Function Calling`）
- 固定流程编排

所以更准确的关系应该是：
- **AI 应用开发** 是上位概念
- **Agent 开发** 是其中一个重要分支

这样定义的好处是：
- 不会一上来就被某个热门名词绑死
- 更贴近真实岗位和真实产品形态
- 也更容易把简单 AI 集成、RAG、工作流、Agent 放进同一张地图里

---

## 为什么这个方向适合传统 Web 开发者
因为 AI 应用开发大多数时候不是脱离传统系统单独存在的，
而是建立在完整的软件工程能力之上。

一个真实的 AI 产品，除了模型调用之外，通常还要处理：
- 用户系统
- 权限控制
- 会话状态
- 文件上传
- 数据存储
- 检索索引
- 异步任务
- 监控告警
- 成本控制
- 配额与计费
- 审计与安全
- 前后端交互体验

所以，模型只是新的能力源，
产品本身仍然是一个完整的软件系统。

这也是为什么传统 Web 开发者在这个方向上其实有天然优势：
- 你已经熟悉系统设计、接口、数据库、缓存、异步、部署这些工程问题
- 你要补的不是“从零变成算法研究员”
- 而是在原有工程能力上，再叠加一层 AI 应用工程能力

---

## 当前建议的能力理解
如果把这个方向压缩成一句话，
我更推荐这样理解：

**AI 应用开发 = 传统工程能力 + AI 应用工程能力**

这里特意不用“Agent 开发能力”来代表全部，
因为 Agent 只是其中一部分。

不过，这里还要补一个很重要的修正：
**这不是简单的“旧底座 + 新能力”叠加，AI 的引入还会反过来改变传统工程的一些做法。**

比如：
- 传统接口大多是确定性的，而 AI 输出天然带有不确定性，这会影响错误处理、测试策略和前端交互设计
- 传统系统的成本大头往往在基础设施，而 AI 系统的成本经常显著受 token 消耗影响，这会反过来影响缓存策略、模型选型和调用路径设计
- 传统可观测性更关注 metrics / logs / traces，而 AI 系统还要补上 eval、质量监控和结果抽检，才能回答“系统到底算不算正常工作”

所以更准确地说，这个方向更像是：
**传统工程能力 + AI 应用工程能力 + AI 对传统工程方式的改造。**

### 1. 传统工程能力仍然是底座
这部分能力包括：
- 后端 / 前端 / 全栈基础
- 数据库与缓存
- API 设计
- 异步任务与消息机制
- 稳定性与容错
- 可观测性
- 权限、安全、部署

如果这部分能力不够强，
最后很容易只能做出“会调用模型接口的 demo”，
而做不出稳定、可维护、可演进的产品。

### 2. 新增的是 AI 应用工程能力
这部分能力至少包括：
- Prompt 设计
- Context Engineering
- 数据准备与知识管理
- Structured Output
- RAG
- Embedding / Retrieval / Re-ranking
- 工具调用
- Workflow 编排
- Agent 设计
- Eval / 评测
- Guardrails / 安全边界
- 成本、延迟、质量取舍
- AI 结果的调试与可观测性

这里面有一块值得单独点出来，因为它很容易被忽略：
**数据准备与知识管理**。

它通常包括：
- 文档解析、清洗、切分
- Embedding 生成与更新
- 向量库或知识库的选型与维护
- 知识更新频率、失效策略和版本管理

这部分能力既不是传统 CRUD 能力的简单延伸，也不能完全被 RAG 这个词替代。
很多 AI 应用能不能做好，关键不只是“模型调得好不好”，而是“喂给模型的上下文和知识到底是不是可用、可维护、可更新的”。

所以更准确地说：
**接下来要补的，不是单独的 Agent 开发能力，而是更上位的 AI 应用工程能力。**

## 当前边界：哪些内容先不作为主线
为了避免方向一开始就过散，当前建议先把边界收清楚。

### 当前主线优先关注
- 在传统产品中集成 AI 能力
- RAG、工作流、工具调用、Agent 等应用层能力
- AI 和业务系统、数据系统、权限系统、监控系统的结合
- AI 功能在真实产品里的质量、成本、延迟和可维护性问题

### 当前不作为主线
- 基础模型训练
- 大规模分布式训练
- 底层模型架构研究
- 推理框架底层优化
- 非常重的算法调优路线

这不是说这些不重要，
而是说：**它们不是当前最现实、最快形成产出的切入点。**

### 一个需要单独说明的灰色地带：Fine-tuning
微调不完全等于模型研究，但它也不属于当前最核心的应用主线。

当前更推荐的判断是：
- **微调会作为一种可选手段被讨论**
- 但**不把它作为 AI 应用开发的主线路径**

更现实的问题通常不是“我会不会微调”，
而是：
- 什么时候 Prompt / Structured Output / RAG 已经够了
- 什么时候确实因为格式稳定性、风格一致性、领域适配等原因，才值得考虑微调

也就是说，微调更适合在后续专题层里作为“判断型方案”出现，
而不是当前一开始就展开的主轴。

这一方向后续如果继续整理，我建议直接复用 `领域知识整理方法论.md` 里的结构。

### 第 1 层：核心原则

### 第 2 层：能力地图
再回答：
- 一个做 AI 应用的工程师，实际要具备哪些能力域

这一层当前已经落到：
- `docs/ai-application/AI应用开发能力地图.md`

当前正式写法采用 9 个能力域，并额外明确了“前置判断 -> 核心构建 -> 能力升级 -> 持续运行”的分层阅读方式。

### 第 3 层：演进线
再回答：
- 一个 AI 应用通常是怎么一步步从简单调用演进到复杂系统的

这一层当前已经落到：
- `docs/ai-application/AI应用开发演进线.md`

当前正式写法已经明确：
- 这条线是常见演化路径，不是必须打满的升级树
- Workflow 通常先于高自主 Agent 成熟落地
- 评测、观测、成本和安全不是最后才补，而是中前期就逐步补
- 单 Agent / 多 Agent 只在前面方案不够时才进入主线

### 第 4 层：专题展开
最后再分别进入具体专题，比如：
- Prompt 与 Context Engineering
- RAG
- 工具调用与 Workflow
- Agent 设计
- Eval 与观测
- 成本 / 延迟 / 质量取舍
- Guardrails 与安全边界
- AI 功能和传统系统集成

## 这个领域的特殊点：变化快，应该怎么写才不容易很快过时
AI 方向和后端有一个明显不同的地方：它变化非常快。

半年前流行的框架、术语和最佳实践，今天可能就已经不是最优解了。
所以这一方向在整理时，要比后端更刻意地做一件事：
**把稳定问题和易变实现分开。**

当前更建议这样处理：
- 核心原则层尽量写稳定约束，比如成本、延迟、不确定性、上下文窗口、评测和安全边界，而不是绑定某个具体框架
- 能力地图层尽量按工作面拆，而不是按某个热门工具拆
- 演进线层尽量关注“问题结构”和“能力升级”，而不是只写工具名称时间线
- 专题层要接受更频繁的更新和重写，因为这一层最容易受生态变化影响

这里有一个很典型的例子：
- 有的生态说 `tool use`
- 有的生态说 `tool calling`
- 也有很多官方文档和历史资料继续使用 `function calling`

从工程视角看，它们大多都在讨论同一类问题：  
**怎样让模型按约定去请求外部能力，再由系统执行并把结果返回给模型。**

所以这套文档里，后续默认优先使用“工具调用”这个中文统称；
第一次出现时，再补充不同生态下常见的英文叫法。

这样做的目的不是追求永不过时，
而是尽量让高层结构比具体工具活得更久。

## 当前已经确认下来的判断
到目前为止，这个方向至少已经固定了下面几个判断：

1. **“AI 应用开发”应该作为上位概念，而不是一开始就把范围缩成“Agent 开发”。**
2. **对于传统 Web 开发者来说，最现实的切入点是 AI 应用工程，而不是模型研究。**
3. **AI 应用工程能力应该被理解为“传统工程能力之上的新增能力层”，而不是一套完全独立的新栈。**
4. **后续继续按“方向判断 -> 核心原则 -> 能力地图 -> 演进线 -> 专题展开”的结构推进，比直接进入框架专题更稳。**

这几个判断现在已经不只是“准备验证的假设”，
而是已经体现在当前目录下的正式文档结构中。
