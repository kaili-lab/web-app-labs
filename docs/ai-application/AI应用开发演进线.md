# AI 应用开发演进线
> 这份文档不讲某个框架怎么用，  
> 而是回答一个更贴近真实项目的问题：  
> **一个 AI 功能，通常是怎么从“接一个模型接口”，逐步演进成“有知识接入、工具调用、工作流、Agent、评测和治理”的系统的？**
>
> 这份文档的目标，是把前面 `核心原则` 和 `能力地图` 里的内容，重新放回一条更线性的工程主线里。

---

## 先约定这份文档写给谁
这份文档默认写给：
- 有传统 Web 开发经验
- 做过一些模型接口调用或小型 AI 项目
- 但还没有经历过复杂 AI 系统演进的程序员

所以这里不会一上来就把你当成架构师，
而是尽量用“一个普通工程师在真实项目里会看到什么问题”的视角来讲。

---

## 先说一个很重要的提醒
这份“演进线”不是在说：
- 所有项目都会走到最后
- 越后面的阶段就一定越高级
- 只要跟着热词往后堆，系统就会越来越好

更接近现实的说法是：

1. **很多有价值的 AI 产品，停在前 4 到 6 个阶段就已经足够好用。**
2. **工作流通常先于高自主 Agent 成熟落地。**
3. **评测、观测、成本和安全不是最后才补，而是从中前期就要逐步补。**

也就是说，这条线更像“常见演化路径”，  
不是“必须打满的升级树”。

---

## 为什么这条线要这样排
如果只看行业热词，很容易误以为 AI 项目会按下面这个顺序发展：

```plain
大模型 -> Agent -> 多 Agent -> 成熟系统
```

但现实里真实项目更常见的顺序，反而是：

```plain
先证明场景成立
  -> 先做最小可用能力
  -> 再补输出约束和上下文
  -> 再补外部知识
  -> 再补工具和工作流
  -> 再补评测、观测、成本、安全治理
  -> 最后只在确实需要时引入 Agent
  -> 只有在单 Agent 不够时才考虑多 Agent
```

这样排的原因很简单：
- 越往后，自主性越高，但复杂度、成本和排障难度也越高
- 很多问题在前面的低复杂度阶段就已经能解决
- 官方最佳实践这两年也越来越强调：**先从最简单、最可控的方案开始，不够用再加复杂度**

---

## 先给出整条主线
你可以先把 AI 应用开发的常见演进线粗略理解成下面这样：

```plain
阶段 0：先判断这是不是一个值得用 AI 的问题
  -> 阶段 1：单次模型调用
  -> 阶段 2：结构化输出
  -> 阶段 3：多轮上下文与会话状态
  -> 阶段 4：外部知识接入（RAG / File Search / Grounding）
  -> 阶段 5：工具调用
  -> 阶段 6：固定工作流编排
  -> 阶段 7：评测、观测、成本与安全治理强化
  -> 阶段 8：单 Agent
  -> 阶段 9：多 Agent / 复杂协作系统
```

下面按这条线逐步展开。

---

## 阶段 0：先判断这是不是一个值得用 AI 的问题
### 先看表面
很多项目一开始最容易犯的错，不是模型选错了，  
而是**根本不该用 AI 的地方也硬上 AI**。

比如有些需求：
- 其实只是固定规则匹配
- 其实只是搜索和筛选
- 其实只是表单自动化
- 其实只是普通工作流自动化

这类问题如果用规则就能稳定解决，
那 AI 很可能只是在引入新的不确定性和成本。

### 往深一层看
AI 最适合的，通常不是“完全确定、规则清晰”的任务，
而是下面这些更偏半结构化的问题：
- 理解自然语言
- 生成内容草稿
- 从长文本里提取结构化信息
- 在模糊需求下给出候选方案
- 在多个候选动作之间做语言层面的判断

所以真实项目里，第一步不是“接哪个模型”，
而是先问：
- 这个任务的核心难点到底是什么？
- 没有 AI 时，现在是怎么做的？
- 当前痛点是人工成本高、理解难、交互差，还是规则根本写不完？

### 先记住一句话
**AI 不是默认解，而是一种在“不好完全写死规则”时才值得引入的新能力。**

---

## 阶段 1：单次模型调用
### 先看表面
这是大多数 AI 项目的起点。

系统最开始通常只是把模型接进现有产品，
做一些非常直接的能力：
- 摘要
- 改写
- 分类
- 情感分析
- 文案生成
- 简单问答

从工程上看，这时的系统还很像一个普通接口：

```plain
用户输入
  -> 后端调用模型
  -> 返回一段文本或一个简单结果
```

### 往深一层看
这一步的核心价值，不是“做复杂”，
而是**先验证这个场景值不值得引入 AI**。

因为很多项目在这一步就会看清几个关键事实：
- 用户到底是不是真的需要这个能力
- 质量是否达到“可用”而不是“惊艳 demo”
- 延迟和成本是否能接受
- 当前模型能力边界在哪里

这也是为什么最新官方实践越来越强调：
**先从最小可用链路开始，不要一上来就做 Agent。**

这里还有一种非常常见的早期形态：  
**模型不直接产出最终内容，而是先做一次轻量判断。**

比如：
- 用户输入 -> 模型判断意图是“查订单 / 退货 / 咨询” -> 进入不同固定分支
- 用户上传文档 -> 模型判断文档类型 -> 进入不同解析链路

这种“模型作为分类器 / 路由器”的模式，  
本质上仍然属于早期轻量形态：  
先用模型补足语言理解，再把后面的执行尽量交给确定性流程。

### 先记住一句话
**单次模型调用阶段的任务，不是追求高级，而是先证明“这个场景成立”。**

### 单词模型调用和agent区别
1. **最小可用链路**的典型形态：用户输入 → 固定 Prompt → LLM 输出 → 固定后处理。甚至可以简化成一个带有良好 Prompt 的单次调用。

2. **Agent** 的本质特征是：LLM 在运行时自主地决定"要不要用工具、用哪个工具、用完之后继续做什么"，存在一个反馈循环（loop）

---

## 阶段 2：结构化输出
### 先看表面
当 AI 输出不再只是给人看，而是要被系统继续处理时，
自然语言结果就开始不够用了。

比如你会很快遇到这些需求：
- 从合同中抽取字段
- 把用户意图分类到固定标签
- 让模型返回可直接渲染的卡片数据
- 让后端根据模型结果走下一步业务流程

这时你需要的就不只是“回答得像”，
而是“输出得可解析、可校验、可消费”。

### 往深一层看
这一阶段真正引入的，不只是一个 JSON 格式，
而是**把 AI 输出从“文本”变成“受约束的接口结果”**。

一旦进入这一步，程序员看待模型的方式就会开始变化：
- 你会关心 schema 是否稳定
- 你会关心字段缺失、类型错误、枚举越界
- 你会关心失败重试和兜底策略
- 你会关心“这个输出到底能不能安全接进业务系统”

也就是说，AI 开始从一个“会写字的能力”，  
变成一个“半稳定的系统组件”。

### 先记住一句话
**AI 输出一旦要进入程序逻辑，就必须尽量从自由文本收紧到结构化结果。**

---

## 阶段 3：多轮上下文与会话状态
### 先看表面
当产品从“单次问答”走向“持续对话”时，
新的问题就出现了：
- 模型要不要记住前面说过什么？
- 用户改口后，系统怎么跟上？
- 历史消息越堆越多，效果为什么反而变差？

这时你会开始接触：
- 会话历史
- 上下文裁剪
- 摘要压缩
- 长对话记忆
- 用户状态与任务状态分离

### 往深一层看
从阶段 2 走到阶段 3，真正的驱动力是：  
**当产品从“单次任务”变成“持续交互”时，前一步那种单次调用、单次返回的无状态模式就不够了。**

这一阶段最重要的转变是：
**你开始意识到，上下文是稀缺资源，不是“塞得越多越好”。**

程序员最容易踩的坑之一，就是把历史聊天全量塞给模型。
结果往往是：
- 成本上升
- 延迟变慢
- 噪声增多
- 关键指令被淹没
- 输出质量反而下降

所以真正重要的不是“记住全部”，
而是“让模型在当前回合看到最有用的信息”。

### 先记住一句话
**多轮对话的本质，不是无限记忆，而是有限上下文下的信息管理。**

---

## 阶段 4：外部知识接入
### 先看表面
做着做着，你会发现一个很现实的问题：

模型也许很会说，
但它并不知道：
- 你的企业文档
- 最新产品规则
- 用户私有数据
- 实时业务状态
- 刚更新的内部知识

这时系统就会走到外部知识接入阶段。
常见形态包括：
- 把少量稳定知识直接写进 system prompt
- 用 few-shot examples 提供格式示范或少量领域规则
- 利用长上下文窗口直接放入少量文档
- 自建 RAG
- 平台内置的 `file search` / `grounding`
- 文档检索 + 重排
- 对数据库或搜索系统做只读查询封装

### 往深一层看
很多程序员一开始会把这一步理解成：
“给模型补点资料”。

但更准确的理解是：
**你开始建设一条“把正确知识在正确时机送进上下文”的链路。**

这里要特别避免一个误区：  
**不是所有知识接入都要立刻上 RAG。**

如果知识量很小、更新不频繁、文档数量也不多，  
更轻量的方式往往已经够用。  
只有当前面这些方式开始扛不住知识规模、更新频率或检索精度要求时，  
RAG 才真正变成更自然的下一步。

这一阶段真正难的通常不是“接一个向量库”，
而是这些问题：
- 文档怎么解析和切分
- 什么知识值得进索引
- 知识更新后怎么刷新
- 召回很多内容时，哪些该留下
- 检索命中了，但为什么答案还是不好

所以这里的重点不是某个具体数据库，
而是“知识供给链路”开始出现。

### 先记住一句话
**很多 AI 问题表面像模型不聪明，本质上是知识没有被正确送进来。**

---

## 阶段 5：工具调用
### 先看表面
再往前一步，产品开始不满足于“只会回答”，
而是希望 AI：
- 查订单
- 查库存
- 调日历
- 调搜索
- 写工单
- 发通知
- 调内部 API

这时就进入了工具调用阶段。

不同生态对这类能力的叫法并不完全一致：
- `tool use`
- `tool calling`
- `function calling`

但从工程视角看，它们大多都在说同一类事：  
**模型负责提出工具请求，系统负责真正执行，并把结果回传。**

### 往深一层看
这一阶段的本质，不是“让模型直接接管系统”，
而是**让模型在边界内获得“查”和“做”的能力**。

这时程序员要开始考虑的东西会明显增多：
- 工具接口如何描述，模型才更容易用对
- 参数 schema 怎么定义
- 哪些动作可以自动执行，哪些必须人工确认
- 失败后怎么重试、回滚或降级
- 工具越来越多时，是否要引入统一连接器或 `MCP` 这类协议

也就是说，这一步开始把 AI 从“语言能力”推向“系统能力”。

### 先记住一句话
**工具调用不是让模型拥有权限，而是让系统按权限边界借给模型能力。**

---

## 阶段 6：固定工作流编排
### 先看表面
当一个任务不再能靠“一次调用 + 一次工具”解决时，
系统会自然走向多步流程。

从阶段 5 走到阶段 6，真正的驱动力是：  
**当一个任务开始需要多次工具调用，而且这些步骤之间存在顺序依赖、状态传递和失败恢复问题时，单次工具调用模式就不够稳定了。**

例如：
- 先识别任务类型，再走不同链路
- 先拆子任务，再并行执行
- 先生成草稿，再做一次检查
- 先检索知识，再调用工具，再汇总答案

这时你会开始看到这些模式：
- Prompt Chaining
- Routing
- Parallelization
- Evaluator-Optimizer
- Orchestrator-Workers

这些在最新官方资料里，通常都更接近 **workflow**，
而不是高自主 Agent。

### 往深一层看
这一阶段真正的变化是：
**你开始用“拆解和约束”来换取稳定性。**

因为很多复杂任务并不是必须交给一个高自主 Agent 自己想办法，
而是可以拆成几步更稳定的子任务：
- 哪一步该固定
- 哪一步该并行
- 哪一步该让模型做判断
- 哪一步该让程序做硬规则校验

现实里，很多能上线、能维护的 AI 系统，
成熟形态其实就停在这一层。

### 先记住一句话
**能用固定工作流解决的问题，通常先不要急着上高自主 Agent。**

---

## 阶段 7：评测、观测、成本与安全治理强化
### 先看表面
做到前面几步之后，系统往往已经“能跑”了。

但这时另一个问题会很快冒出来：
- 这个版本真的比上一个版本更好吗？
- 为什么线上都返回 200，用户还是觉得不好用？
- 为什么成本一上量就爆了？
- 为什么偶发越权、幻觉、Prompt Injection 开始变成真实风险？

这时评测、观测、成本和安全治理就会从“配角”变成“主角”。

### 往深一层看
这一步很容易被误解成“最后补的运维工作”，
但其实更准确的说法是：

**这些能力从前面几个阶段就已经在出现，只是到这里变成了系统能不能继续迭代的关键前提。**

也可以更直白地理解成：  
**阶段 7 不是“开始做治理”，而是“治理从配角升级为主角”的转折点。**  
在此之前，你也会做一些轻量治理；  
但走到这里之后，如果治理跟不上，就不应该继续往更高自主性推进。

你会开始系统性地补这些东西：
- 离线评测集
- 回归测试
- Trace / 轨迹观测
- 失败样本回放
- 线上抽检
- token 与延迟监控
- 配额与限流
- 敏感信息处理
- 审计日志
- 高风险动作的人类确认

没有这些能力，系统也许能演示，  
但很难持续演进。

### 先记住一句话
**AI 系统从“能跑”到“能上线、能迭代”，中间差的往往不是模型，而是治理能力。**

---

## 阶段 8：单 Agent
### 先看表面
只有当前面的固定流程真的不够用时，
项目才会自然走到单 Agent 阶段。

典型场景通常是：
- 任务路径很难预先写死
- 需要多轮规划、观察、再行动
- 同一个目标下，可能有多种中间路径
- 工具使用顺序需要运行时决定

比如：
- 开放式研究任务
- 较复杂的代码修复或分析任务
- 跨多个系统的故障排查助手
- 需要“边看边做边调整”的操作型任务

### 往深一层看
单 Agent 的本质，可以先粗略理解成一个循环：

```plain
理解目标
  -> 规划下一步
  -> 选择工具
  -> 执行动作
  -> 读取结果
  -> 决定继续还是停止
```

这一步带来的不是“更高级”，  
而是明显更高的系统复杂度：
- 状态更多
- 调试更难
- token 消耗更高
- 失败路径更长
- 结果更难预测

所以现实里，Agent 不是默认解，  
而是“固定工作流已经不够描述任务路径”时才值得付出的代价。

### 先记住一句话
**Agent 适合开放任务，不适合拿来包装本来就能写死的流程。**

---

## 阶段 9：多 Agent / 复杂协作系统
### 先看表面
再往后，只有当单 Agent 也开始不够时，
才会出现多 Agent。

常见原因通常不是“想显得更先进”，
而是下面这些更现实的问题：
- 一个 Agent 上下文装不下所有任务信息
- 不同子任务需要不同专长
- 某些步骤适合并行
- 需要把计划、执行、审查分给不同角色

这时系统可能会变成：
- 一个总控 Agent 负责拆任务
- 多个专长 Agent 分别处理子问题
- 某个审查 Agent 负责验收或复核

对大多数个人项目和中小团队来说，  
走到阶段 6 到 7，往往已经能覆盖绝大部分真实需求。  
阶段 8 到 9 更多出现在平台级产品、复杂内部系统或大型企业场景里。

### 往深一层看
多 Agent 的真实价值主要有三类：
1. **分工**：把不同专长隔离开
2. **并行**：缩短大任务总耗时
3. **上下文隔离**：减少一个大上下文里信息互相污染

但它引入的新复杂度也非常直接：
- Agent 之间怎么通信
- 谁对最终结果负责
- 某个子 Agent 出错时如何定位
- 如何做全链路评测
- 如何防止系统只是在“自我对话”中空转

所以多 Agent 不是单 Agent 的自然升级版，
而是当单 Agent 的职责边界、上下文容量和执行效率都开始成为瓶颈后，才有意义的结构化拆分。

### 先记住一句话
**多 Agent 的核心价值不是“更像人类组织”，而是更好的分工、并行和上下文治理。**

---

## 现实里还有一条“横着长出来”的配套线
如果只看上面的主线，你可能会误以为：
- 先做功能
- 最后再补评测和治理

但真实项目更像是两条线同时在长：

```plain
主能力线：
单次调用 -> 结构化 -> 上下文 -> 知识 -> 工具 -> 工作流 -> Agent

配套治理线：
日志/追踪 -> 评测 -> 成本控制 -> 安全边界 -> 权限/审计 -> 人工介入
```

区别只是：
- 前期这些治理能力可能比较轻
- 到中后期它们会变成主角

所以更准确地说，AI 应用开发并不是只会“加能力”，
还会不断要求你“补治理”。

---

## 这条演进线和前两份文档是什么关系
- `核心原则`回答：AI 应用开发为什么会复杂
- `能力地图`回答：做 AI 应用的人平时在解决哪些类问题
- `演进线`回答：这些问题通常会按什么顺序、在什么压力下出现

所以它们不是重复，而是三个不同视角：
- 核心原则：看底层约束
- 能力地图：看工作面
- 演进线：看时间顺序和系统成长路径

---

## 最后给一个更贴近现实的判断
如果你以后在真实项目里看到一个 AI 功能，
更合理的判断顺序通常不是：

```plain
这个需求要不要做 Agent？
```

而更应该是：

```plain
这个问题到底适不适合用 AI？
  -> 单次调用够不够？
  -> 输出要不要结构化？
  -> 需要补外部知识吗？
  -> 需要接工具吗？
  -> 能否用固定工作流解决？
  -> 是否已经到了必须上 Agent 的程度？
  -> 评测、观测、成本、安全现在补到什么程度才够？
```

当你开始按这条路径思考时，
你对 AI 应用开发的理解，就已经不再只是“会调模型接口”，
而是开始具备真正的工程判断力。
