# AI 应用开发目录

这里记录 AI 应用开发方向的初步判断、知识拆分思路和后续整理入口。

## 默认读者
这里的文档默认写给：
- 有传统 Web 开发经验
- 做过一些模型接口调用或小项目
- 但还没有系统建立 AI 应用开发知识结构的程序员

所以这里的写法会优先从工程直觉和产品落地角度解释问题，
而不是默认读者已经熟悉算法研究或复杂 Agent 理论。

## 术语写法原则
AI 领域术语变化很快，所以这个目录默认采用一个更稳妥的写法：
- 优先使用面向工程理解的中文统称
- 第一次出现时，再补常见英文叫法
- 尽量避免把文档绑死在某一家厂商当下的命名上

例如：
- 工具调用：对应 `tool use` / `tool calling`，很多资料里也会写作 `function calling`
- 工作流：对应 `workflow`

这样写的目的，是让文档先帮你抓住“它本质在解决什么问题”，
再去理解不同厂商或框架各自怎么命名。

## 这一类文档的职责
这一目录负责回答：
- 对传统 Web 开发者来说，AI 方向最现实的切入点是什么
- 当前应该把范围界定在什么地方，哪些内容先不作为主线
- AI 应用开发后续应该怎样按“核心原则 -> 能力地图 -> 演进线 -> 专题展开”继续整理
- 这一方向和现有后端工程能力之间是什么关系

它的重点不是马上进入某个框架或 API 的细节，
而是先把“这个方向到底在学什么、为什么值得学、应该怎么拆”讲清楚。

## 这一类文档不负责什么
- 不负责展开模型训练、底层推理优化、分布式训练等研究型内容
- 不负责替代具体 AI SDK、框架、平台的官方文档
- 不负责立即进入某个 Agent 框架的操作手册

## 和其他文档的关系
- 如果你想看跨领域通用整理方法 -> 回到 `领域知识整理方法论.md`
- 如果你想看当前对 AI 应用开发方向的总体判断 -> 看当前目录
- 如果你想看后端方向已经整理出的示范结构 -> 看 `docs/java-topics/`、`docs/capability-map/`、`docs/roadmap/`

## 当前包含的文档
- `docs/ai-application/AI应用开发方向判断.md`：记录当前对 AI 应用开发方向的范围、价值和知识拆分建议
- `docs/ai-application/AI应用开发核心原则.md`：提炼 AI 应用开发背后最稳定的约束和根节点
- `docs/ai-application/AI应用开发能力地图.md`：把核心原则翻译成 AI 应用工程师实际要具备的能力块
- `docs/ai-application/AI应用开发演进线.md`：把这些能力和约束放回真实项目常见的演化顺序里

## 当前建议的使用方式
1. 先读 `领域知识整理方法论.md`
2. 再读 `docs/ai-application/AI应用开发方向判断.md`
3. 再读 `docs/ai-application/AI应用开发核心原则.md`
4. 再读 `docs/ai-application/AI应用开发能力地图.md`
5. 再读 `docs/ai-application/AI应用开发演进线.md`
6. 在此基础上继续决定下一步要展开哪个专题
