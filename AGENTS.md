# Work Style
- 开始改动前先理解现有结构和上下文
- 聚焦当前任务，不主动重构、优化或修复无关问题

- 修改已有内容时，尽量保持 diff 小而清晰
- 遇到会影响仓库结构的决策，先给出选项和影响

# Teaching Preferences
When I ask you to explain a concept or code:
1. 先说明它属于哪个领域、在解决什么问题、和更大知识网络的关系
2. 再解释它做什么、为什么存在、为什么当前语境这样使用
3. 每次只深入一层，等我继续追问
4. 优先用生活类比帮助理解
5. 讲代码模式时给最小示例，不给整套实现

# Project Instructions

## Session Start
- 每次新会话先读 `README.md`、`项目记忆.md`、`领域知识整理方法论.md`
- 进入某条主线前，再读对应目录的 `README.md`，例如 `docs/fullstack/README.md`、`docs/ai-application/README.md`
- 如果用户明确把 `references/` 下的材料当作输入，再去读对应文件

## Documentation Rules
- 所有项目文档使用中文
- 写任何文档前，先明确：文档对象、适用边界、抽象层级、当前讨论的是目标形态还是首版范围
- 不要把别的仓库的结构和口径直接继承到当前仓库；要先判断是否符合本仓库目标
- 新文档要和 `项目记忆.md` 保持一致；如果目标、结构或重大决策变化，同步更新 `项目记忆.md`

## Repo Direction
- 这个仓库的一级主线是“现代应用 / 全栈认知”，不是技术栈清单
- `React`、`Next.js`、`Node.js` 等默认是实现手段，不自动上升为一级分类
- `AI` 是重点专题线，和主线紧密关联；如需独立展开，必须写清它服务的应用问题和边界
- 文档优先回答“为什么会出现、解决什么问题、代价是什么”，实验再回答“代码如何落地、运行时会发生什么”
- 涉及 React、Next.js、AI 等变化较快的技术栈时，默认优先参考最新稳定版；如果版本信息会影响判断，先核查官方资料
