# 项目记忆

## 项目目标
构建一个面向现代应用开发的学习型项目，帮助把零散的前端、后端、架构、产品与 AI 相关知识，整理成一套可逐层缩放的认知体系。

这里的“逐层缩放”指的是：
- 先从 high level 看到一整个现代应用会遇到哪些问题
- 再理解这些问题为什么出现、常见方案是什么、代价是什么
- 最后再按需要下钻到某个具体专题内部

## 目标使用者
面向已经有一定开发经验，但希望把零散经验串联成现代全栈认知体系的程序员。

默认读者通常：
- 做过一些前端或后端开发
- 知道不少技术名词，但连接关系还不够稳定
- 希望建立“问题 -> 方案 -> 取舍 -> 演进”的整体判断力

## 核心学习原则
只有当应用出现具体需求、复杂度或压力时，才引入对应技术。

这意味着：
- 不为了“流行”提前堆技术
- 不为了“显得完整”提前搭大而全基础设施
- 优先建立“问题 -> 分析 -> 方案 -> 取舍”的判断路径
- 先理解为什么需要它，再理解怎么实现它
- high level 先行，但不排斥后续深入底层

## 当前核心判断
`web-app-labs` 不是技术栈拼盘仓库，也不是脚手架仓库。

当前共识是：
- 仓库一级主线是“现代应用 / 全栈认知”
- `AI` 是重点专题线，不默认与仓库一级主线并列
- `React`、`Next.js`、`Node.js` 等是常见实现手段，不是一级知识主线
- 同一知识点应尽量回答：它解决什么问题、在什么场景值得引入、会带来什么代价

## 为什么这样组织
这样组织的原因是：
- 现代应用涉及前端、服务端、数据、协作、部署、AI 等多个方向，直接逐个深挖很容易再次碎片化
- 先建立全局地图，才能知道一个知识点处于什么位置、和什么问题相连
- 当后续进入某个专题时，能够更清楚地知道“为什么值得往下钻”

## 当前文档组织原则
当前项目以三类内容并行：
- 仓库级文档：负责定义目标、结构边界和整理方法
- 主线 / 专题文档：负责串联“问题 -> 分析 -> 方案 -> 取舍 -> 演进”
- 实验代码：负责验证某个方案在代码和运行时里的真实行为

当前共识是：
- 文档先回答“为什么这样设计、边界在哪里、什么时候需要它”
- 实验再回答“怎么落地、能观察到什么现象”
- 不让实验代码替代知识结构，也不让知识文档脱离真实应用场景

## 当前目录方向
- `README.md`：仓库级概览
- `项目记忆.md`：仓库目标、主线边界、长期共识
- `领域知识整理方法论.md`：跨领域可复用的知识整理方法
- `docs/fullstack/`：现代 Web / 全栈主线文档
- `docs/ai-application/`：AI 应用专题线文档
- `labs/`：代码实验、原型与验证项目

## 目录职责边界
- `docs/fullstack/` 负责回答：
  - 一个现代 Web 应用从简单到复杂会遇到哪些问题
  - 前端、后端、数据、协作、部署等问题如何被串联起来
  - 每类能力为什么重要、什么时候值得引入、边界在哪里

- `docs/ai-application/` 负责回答：
  - AI 能力会在现代应用里解决什么问题
  - AI 应用开发与传统应用工程能力如何衔接
  - AI 相关方案的核心取舍、边界与演进线是什么

- `labs/` 负责回答：
  - 某个方案跑起来是什么样
  - 能观察到什么现象
  - 不同实现方式会带来什么差异
  - 当前判断在代码层面是否成立

## 抽象层级约定
- 一级结构优先按“问题域 / 主线”组织，而不是按技术名词罗列
- 技术栈是承载应用问题的实现手段，不是默认顶层分类
- 如果某条线只是某个主线下的专题，必须明确写出它服务的对象和边界
- 同一个主题在不同缩放距离下，看到的问题不同；写文档时要先说明当前站在哪个观察层级

## 当前决定
- 采用“仓库级方法论 + 主线 / 专题文档 + 代码实验”的结构
- 一级主线先固定为“现代应用 / 全栈认知”
- AI 内容继续保留并展开，但默认作为重点专题线挂接在主线旁边理解
- 暂不为了未来可能的复杂度，提前引入 monorepo、大而全脚手架或大量空目录
- `docs/fullstack/` 下已有按技术栈拆分的材料，当前视为过渡性笔记，不代表最终一级结构
- 快速变化的 AI 术语和能力边界，如涉及最新事实，优先核查官方资料后再落文档
- 每次确认新的方向、结构或整理方式后，都要回看 `README.md`、`AGENTS.md`、`项目记忆.md`

## 文档一致性复查约定
- 先确认文档对象：是在写整个仓库、某条主线、某个专题，还是某个实验项目
- 先确认“阶段”含义：这里说的是知识演进阶段、产品复杂度阶段，还是代码实施阶段
- 关键决策优先只在 `项目记忆.md` 定稿，其他文档尽量映射和引用，不重复发明新口径
- 每次改动项目结构、主线边界或技术范围后，至少回看 `README.md`、`项目记忆.md` 和对应目录下的 `README.md`
- 如果某条规则只对单个主线成立，必须显式写出适用对象，避免被误写成整个仓库共识

## 当前路线
1. 先把仓库级目标、边界和方法论固定下来。
2. 补齐 `docs/fullstack/` 的主线骨架，优先按问题域而不是技术名词组织。
3. 继续整理 `docs/ai-application/`，并明确它与现代应用主线的连接点。
4. 围绕当前最真实的问题，增加对应实验，而不是预建大而全工程。
5. 随着学习推进持续迭代结构。

## 工作规则
先启动并迭代，不等待完美设计。
保持对象清晰、边界清晰、抽象层级清晰。
避免再次把仓库写成“技术栈清单”或“基础设施搭建记录”。
